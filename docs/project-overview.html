Agent ROI Demo

Measure the real-world value of AI agents by combining telemetry, user feedback, and cost data into a single ROI score.

Cortex Agents Cortex REST API External Agents

What does this demo show?

This project demonstrates how to measure whether AI agents are actually delivering value. It connects multiple types of agents to a unified monitoring system that tracks three things:

⏱

Performance

How long does each step take? Where are the bottlenecks? See every tool call, retrieval, and generation step in a waterfall chart.

💬

User Feedback

Thumbs up/down, star ratings, and task completion tracking. Users tell you if the agent actually helped them.

💰

Cost

Credits consumed per conversation, per tool, per day. Know exactly what you're spending and whether it's worth it.

The ROI Formula (Customizable)

The demo ships with a default formula, but this is designed to be customized for your organization:

// Default: Quality x Reliability / Cost ROI Score = (Positive Feedback Rate x (1 - Error Rate)) / Credits Per Conversation

But real organizations want to quantify value in dollars. Here are three approaches you can plug in:

Option A: Time Saved

Track user role and FTE salary. When a task is completed, multiply time saved by the user's hourly rate.

// User: Senior Analyst, $85/hr // Task: "Generate quarterly report" // Time saved: 30 min Value = $85/hr x 0.5hr = $42.50 Cost = 0.003 credits x $3/credit = $0.009 ROI = $42.50 / $0.009 = 4,722x

Option B: Action Value

Assign dollar values to actions the agent enables (e.g., resolving a support ticket, closing a deal, approving an order).

// Rate table: // Support ticket resolved: $15 // Data report generated: $25 // Policy question answered: $5 Value = SUM(action_values) ROI = Total Value / Total Cost

Option C: Deflection Rate

If the agent handles a request that would have gone to a human, the value = the fully-loaded cost of that human interaction.

// Avg support interaction: $12 // Agent handled: 847 conversations // Positive resolution: 78% Value = 847 x 0.78 x $12 = $7,928 Cost = 847 x $0.009 = $7.62 ROI = 1,040x

The feedback system captures the signals needed for all three approaches: time saved (from task completion), action type (from categories), and resolution quality (from thumbs up/down).

Three production agents, three architectures

The demo includes three user-facing agents built with different technologies to show that the monitoring system works regardless of how the agent is built.

Why three architectures?

Cortex Agents are fully managed by Snowflake. You get automatic telemetry, built-in tools (Cortex Analyst, Cortex Search), and zero infrastructure to manage. Best for production workloads.

Cortex REST API Agents are custom Python applications that still use Snowflake for LLM inference (via CORTEX.COMPLETE). You get more control over the agent logic while keeping Snowflake as the AI backbone.

External Agents run entirely outside Snowflake. The demo uses a local LLM (no cloud calls for inference). Telemetry is still sent to Snowflake for unified monitoring, but the agent itself is independent.

Optimizing and improving agents

The fourth agent in the system is different from the other three. It doesn't serve end users -- it serves the team that builds and operates the other agents.

Trace Analyst Agent

A Cortex Agent connected to a semantic view over the materialized trace and feedback data. You can ask it natural-language questions like:

  • "What is the average latency for the Sales Agent's planning step?"
  • "Which tool has the highest error rate this week?"
  • "Show me conversations where users gave thumbs down after a replan"
  • "Compare response time between the Cortex Agent and the Knowledge RAG Agent"

How it works today

The Trace Analyst is embedded in the Traces page as a sidebar chat. It queries materialized tables (TRACE_SPANS_DT, TRACE_FEEDBACK_DT) via Cortex Analyst to answer performance questions. This gives you a conversational interface over what would otherwise require writing complex SQL against observability tables.

Future vision: autonomous agent improvement

Today, the Trace Analyst informs humans who then manually update agents. The next evolution is automating parts of this loop:

Auto-suggest instruction changes

When the Trace Analyst detects recurring negative feedback patterns (e.g., "Hallucination" on questions about shipping), it could automatically draft improved system instructions and present them for human approval.

Example: "I noticed 12 hallucination reports this week on shipping questions. Suggested instruction update: 'Always cite the specific policy document when answering shipping questions. If no document matches, say you don't have that information.'"

Auto-update agent instructions

With appropriate guardrails, the system could apply instruction updates automatically. Use Snowflake's ALTER AGENT to update the agent's system prompt, then monitor whether the ROI score improves or degrades over the next N conversations.

Guardrail: Only apply changes that passed A/B testing in a shadow mode first. Auto-rollback if error rate increases by more than 5%.

Agent world model testing

Before deploying instruction changes to real users, test them against a "world model" -- a synthetic dataset of historical questions and expected behaviors. Run the updated agent through 100+ representative queries and compare outputs against ground truth.

Implementation: Use Snowflake's AI Observability evaluation runs to score the candidate instructions against historical data before promoting to production.

Cross-agent learning

If the Local Q&A Agent handles a question better than the Cortex Agent (based on feedback), the system could analyze what's different and suggest routing rules or instruction improvements for the weaker agent.

Example: "The Local Agent scored 4.2 stars on 'What is photosynthesis?' while the Sales Agent scored 2.1. Consider routing general knowledge questions away from the Sales Agent."

Automated tool selection tuning

When trace data shows an agent frequently picks the wrong tool (e.g., using Cortex Search when Cortex Analyst would be better), the system could suggest tool-routing instructions or even adjust tool descriptions to improve selection accuracy.

Signal: Traces where tool selection is followed by a replan (the agent tried a tool, failed, and had to try another).

Cost optimization recommendations

Identify conversations where a smaller/cheaper model would have produced an equivalent result. Suggest model downgrades for specific query types to reduce cost without impacting quality.

Example: "87% of policy questions are answerable by llama3.1-8b at $0.001/query vs the current model at $0.01/query. Switching would save $2,400/month with no quality impact."

How it works

The system has four main pages, each serving a different audience:

Chat

Talk to any agent. Switch between them with a dropdown inside the message box. Leave feedback with thumbs up/down or track tasks.

Dashboard

See ROI metrics at a glance: conversations, cost per request, positive feedback rate, error rate, and the combined ROI score.

Traces

Inspect individual conversations step-by-step. A Gantt-chart waterfall shows exactly what happened and how long each step took.

Config

Register new agents, configure endpoints, toggle agents on/off. Supports Cortex Agents, REST API agents, and external agents.

The feedback model

The underlying system is simple: you send a JSON object with any fields you want. The API accepts three things:

{ "positive": true, // Was this helpful? (required) "categories": [...], // Tags you define (any strings) "feedback_message": "..." // Free-text comment (optional) }

The categories array is where all the magic happens. You can put anything in there. The UI interprets the categories to build different interfaces, and the backend processes them for ROI calculations.

How the backend processes categories

Categories are just strings. The backend parses them with simple pattern matching:

// Categories follow a key:value convention "stars:4" → stars = 4, used for quality scoring "value:High" → task value = High (weighted 3x in ROI calc) "time_saved:15-30 min" → converted to dollars via user's hourly rate "automated:yes" → automation rate metric (higher = more efficient) "deal:closed_won" → custom metric you define for your use case // The system doesn't care what you put in categories. // Your dashboard/ROI formula reads whatever keys matter to you.

This means you can design any feedback interface for your use case, send it through the same API, and build custom ROI formulas that read your specific category keys. No backend changes needed to add new feedback types.

The process: building an Agent ROI program

This isn't just a technology project. It's a measurement program that requires alignment between product, engineering, and finance. Here's the sequence that works:

Phase 1: Define what "value" means

Before writing any code, answer these questions with your stakeholders:

What outcomes matter?

Is the goal to save employee time? Reduce support ticket volume? Increase deal velocity? Improve data quality? Each outcome has different KPIs and different dollar values.

Who are the users?

An agent that saves a $200/hr executive 10 minutes is worth more than one that saves a $50/hr analyst the same time. Map users to roles, roles to rates.

What does "good" look like?

Define your target ROI threshold. 10x? 100x? 1000x? This determines whether you optimize for cost reduction, value increase, or both.

Phase 2: Design the feedback system

The feedback UI is your primary data collection mechanism. Design it for the signals your ROI formula needs:

If your ROI model needs...Your feedback UI should capture...Example categories
Time savingsHow long the task would have taken manuallytime_saved:15-30 min
Quality assessmentWas the output correct and complete?stars:4, Incomplete
Task completionDid the user actually finish the work?task:complete, automated:yes
Business valueWhat was the dollar impact?value:High, deal:closed_won
Error classificationWhat went wrong and why?Hallucination, Wrong tool used
User effortHow many turns to get the answer?turns:3, needed_edit:yes

Start simple (thumbs up/down), then layer in richer signals as you understand what drives value. You can always add new feedback types without changing the backend.

Phase 3: Choose your agent strategy

This is simultaneously a technology decision, a procurement decision, and an architecture decision. Consider:

Fully managed (Cortex Agent)

Choose when: You want production-ready agents fast. You trust Snowflake's LLM selection. You want automatic telemetry with zero instrumentation effort.

Trade-off: Less control over model choice and reasoning logic. Costs scale with Snowflake credit pricing.

Hybrid (Cortex REST API)

Choose when: You want custom agent logic but still want Snowflake's LLMs. You need specific retrieval patterns or multi-step workflows that Cortex Agent's built-in tools don't support.

Trade-off: More engineering effort. You manage the server, but inference costs stay on Snowflake.

Fully external

Choose when: You have regulatory requirements for on-premise inference. You want to use a specific model (GPT-4, Claude, local LLM). You're building for a future where you might switch providers.

Trade-off: You manage everything. Cost attribution is harder. But you maintain full flexibility.

The flexibility question: Most organizations should plan for a blend. Start with Cortex Agents for speed-to-value, but build the monitoring infrastructure (like this demo) to be agent-agnostic. When a new model or provider becomes compelling, you can add it without rebuilding your ROI measurement system.

Phase 4: Instrument and deploy

Build the technical infrastructure:

  1. Agent registry -- A single source of truth for all agents (type, endpoint, observability config)
  2. Telemetry pipeline -- Ensure all agents emit spans to the same event table (automatic for Cortex, TruLens for external)
  3. Attribute normalization -- Map diverse span formats to a standard schema so dashboards work across agents
  4. Feedback collection -- Deploy the feedback UI designed in Phase 2, connected to the feedback API
  5. Cost attribution -- Connect credit usage, query history, and warehouse metering to agent activity

Phase 5: Measure, iterate, expand

Once live, the cycle is:

Collect traces + feedback
→
Calculate ROI per agent
→
Identify low-ROI agents
→
Tune or replace
→
Measure again

Key questions to ask each cycle:

  • Which agents have the highest ROI? Can we route more traffic to them?
  • Which agents have high error rates or negative feedback? What's failing?
  • Are users actually completing tasks, or just asking questions and leaving?
  • Is the cost per conversation trending up or down? Why?
  • Are there use cases where a cheaper agent (smaller model, local LLM) delivers equivalent value?

Key learnings and recommendations

Findings from building this system, each with a clear action step for product teams.

1. Telemetry spans differ across agent types -- you need a normalization layer

What we found: Cortex Agents emit rich, structured spans with Snowflake-specific attribute names (e.g., snow.ai.observability.agent.planning.model). TruLens-instrumented agents emit spans with different attribute names (e.g., ai.observability.call.kwargs.input). LangChain agents emit yet another format. Without normalization, your monitoring tools can't compare agents side-by-side.

Action step: Define a standard set of telemetry attributes (model, query, tokens, status, response) and maintain a mapping table that translates each framework's native attribute names into your standard. We built SPAN_ATTRIBUTE_MAP for this. Adding a new framework = INSERT rows, no code changes. Recommend this becomes a managed Snowflake feature.

2. Cortex Agents show rich traces; external agents show simpler ones (for now)

What we found: A Cortex Agent trace shows 5-8 distinct span types: planning, tool selection, Cortex Analyst call, Cortex Search call, SQL execution, response generation, and sometimes replanning. External agent traces typically show fewer spans (e.g., just "generate" or "retrieve + generate").

Is this a permanent limitation? No. The difference is because:

  • Cortex Agents have a built-in reasoning loop (plan → select tool → execute → evaluate → respond or replan). Snowflake instruments every step automatically.
  • Our external agents are intentionally simple (single RAG pipeline, or a 3-step LangGraph). They don't have tool selection, replanning, or multi-tool orchestration.
  • If you build an external agent with full agentic behavior (tool router, memory, evaluation loops), TruLens/LangGraph will emit spans for each step -- producing traces just as rich as Cortex Agents.

Action step: The trace richness depends on agent complexity, not the monitoring system. If you need comparable trace depth, instrument your external agents with the same patterns: separate the planning step, tool selection, tool execution, and response generation into distinct functions or nodes. The observability system will capture them.

3. Feedback events have no trace identifier

What we found: When a user submits feedback via the Cortex Agent :feedback API, the resulting observability event has no trace_id or input_id linking it to the specific conversation it's about. We can only match feedback to traces by timestamp proximity (within 60 seconds).

Action step: File a product request for Snowflake to include a trace_id or request_id in feedback events. Until then, design your UI to submit feedback immediately after a response (minimize the time gap), and use conservative matching windows. For external agents, we solved this by storing the TruLens record_id alongside feedback for exact matching.

4. Dynamic tables can't use table functions as sources

What we found: We tried to create a dynamic table over GET_AI_OBSERVABILITY_EVENTS() for auto-refreshing materialized trace data. Snowflake rejected it: table functions are not supported as dynamic table sources.

Action step: Use a scheduled task (CTAS every 5 minutes) as the alternative for materializing observability data. Budget for the warehouse cost of periodic refresh. Alternatively, query the table function directly and accept the latency (~1-2 seconds per query).

5. Credit attribution has 60-minute latency

What we found: CORTEX_AI_FUNCTIONS_USAGE_HISTORY takes up to 60 minutes to reflect new usage. QUERY_HISTORY has up to 45-minute latency. This means your ROI dashboard won't show real-time cost data.

Action step: Design dashboards with this expectation. Show "last updated" timestamps. For real-time cost estimates, consider maintaining a running estimate based on token counts (available immediately in spans) multiplied by published per-token rates, then reconcile with actual credit data after the latency window.

6. TruLens OTEL export requires careful setup

What we found: Three gotchas with TruLens's Snowflake span exporter:

  • Version case sensitivity: External Agent versions must be uppercase (V1 not v1) or the ingestion procedure fails silently.
  • Background evaluator crashes: The evaluator thread throws errors when no prior events exist. Always pass start_evaluator=False.
  • PAT authentication: Use authenticator='programmatic_access_token', not 'oauth'.

Action step: Document these in your team's setup guide. For production deployments, wrap TruLens initialization in try/catch with clear error messages for each failure mode. Consider contributing fixes upstream to the TruLens open-source project.

7. The feedback API is a generic envelope -- not a rigid schema

What we found: The Cortex Agent :feedback API accepts any strings in the categories array. This means you're not locked into "thumbs up/down" -- you can design arbitrarily rich feedback interfaces (star ratings, task completion, deal outcomes, code review scores) and send it all through the same API.

Action step: Treat feedback design as a product decision, not a technical one. Define a key:value convention for categories (e.g., "stars:4", "value:High", "time_saved:15-30 min"). Build your ROI formula to parse whatever keys you define. The backend and API don't need to change when you add new feedback types -- only the UI and the ROI calculation logic.
